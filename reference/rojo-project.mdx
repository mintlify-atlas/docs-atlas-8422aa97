`default.project.json` is the Rojo project file that maps your `src/` directory to the Roblox service hierarchy. genRojoTree writes this file from scratch on every run — it is never hand-edited. The output combines a fixed static skeleton (startup scripts, UI, and external packages) with dynamic entries inserted by the file walker for every `.luau` file it discovers under `src/`.

<Note>
  This file is auto-generated by `tools/genRojoTree.js` and should not be
  edited by hand. Any manual changes will be overwritten the next time you run
  `npm run build:rojo` or when `watch:rojo` detects a file change. To modify
  the structure, update the `tree` constant in the script instead. See the
  [script reference](/reference/script) for details.
</Note>

## Top-level fields

<ResponseField name="name" type="string" required>
  The Rojo project name. Set to `"genrojotree"` in the `tree` constant inside
  `genRojoTree.js`. Rojo uses this as the DataModel name in Studio when the
  project is served or built. Change it to match your actual project by editing
  the `name` field in the `tree` constant.
</ResponseField>

<ResponseField name="tree" type="object" required>
  The root of the Roblox instance tree. The top-level object must contain a
  `$className` of `"DataModel"`. Every key nested inside `tree` corresponds
  to a Roblox service or instance.

  <Expandable title="tree properties">
    <ResponseField name="$className" type="string">
      Declares a virtual Roblox instance. Rojo creates the instance with this
      class name in the DataModel. No file system directory is required.
      Common values are `"DataModel"` (root only) and `"Folder"`.
    </ResponseField>
    <ResponseField name="$path" type="string">
      Points to a file system path (relative to the project root). Rojo syncs
      the file or directory at that path into the instance. A `.luau` file
      becomes a `ModuleScript`, `LocalScript`, or `Script` depending on its
      suffix. A directory with an `init.luau` becomes a `ModuleScript` whose
      children are the other files in the folder.
    </ResponseField>
  </Expandable>
</ResponseField>

## `$className` vs `$path` nodes

Every node in the tree uses one of two strategies:

| Strategy | Key | Purpose |
|----------|-----|---------|
| Virtual folder | `$className: "Folder"` | Creates a Roblox `Folder` instance with no backing file. Children are populated by the walk. |
| File system reference | `$path: "<relative-path>"` | Syncs a file or directory from disk into Roblox. Rojo manages the instance type automatically. |

Static entries like `Packages` and `UI` use `$path` because they point at directories that Rojo should sync wholesale. Dynamic entries discovered by the walk always use `$path` for individual `.luau` files and `$className: "Folder"` for intermediate container nodes.

          "$className": "Folder"

      },

        "$path": "src/ui"
      }
    },
    "ServerScriptService": {
      "Server": {
        "$path": "src/startup/Server.server.luau"
      },
      "Services": {

## Service nodes

### `ReplicatedStorage.Shared`

A virtual `Folder` that holds all client-side and shared scripts discovered by the walk. It contains three dynamic sub-folders that are populated at generation time.

<ResponseField name="Services" type="object">
  A `$className: "Folder"` container. The walk inserts a child folder for each
  service directory found under `src/services/`, then places the individual
  script entries inside it. Client and utility scripts for a service land here;
  the server script goes to `ServerScriptService.Services` instead.
</ResponseField>

<ResponseField name="Classes" type="object">
  A `$className: "Folder"` container for class modules discovered under
  `src/classes/`. Empty when no files are present.
</ResponseField>

<ResponseField name="Modules" type="object">
  A `$className: "Folder"` container for general shared modules discovered
  under `src/modules/`. Empty when no files are present.
</ResponseField>

---

### `ReplicatedStorage.Packages`

```json
"Packages": { "$path": "Packages" }
```

A static `$path` entry pointing at the `Packages/` directory in the project root. This directory is managed by [Wally](https://wally.run), the Roblox package manager, and is not part of the `src/` tree that the walk processes. All Wally-installed packages are synced here automatically.

---

### `ReplicatedStorage.UI`

```json
"UI": { "$path": "src/ui" }
```

A static `$path` entry pointing at `src/ui/`. The entire directory is synced by Rojo as-is. `src/ui` is included in `BLACKLISTED_DIRS` so the walk does not also try to insert individual files from it into the dynamic tree.

---

### `ServerScriptService.Server`

```json
"Server": { "$path": "src/startup/Server.server.luau" }
```

The server-side entry point script. This is a static `$path` node; Rojo syncs it as a `Script` (because of the `.server.luau` suffix). `src/startup` is included in `BLACKLISTED_DIRS` so it is not processed by the walk.

---

### `ServerScriptService.Services`, `.Classes`, `.Modules`

Mirror of the `ReplicatedStorage.Shared` sub-folders for server-side scripts. Any `.luau` file whose base name (lowercased) contains `"server"` is routed here instead of to `ReplicatedStorage.Shared`. In the example output, `ExampleServiceServer` lands under `ServerScriptService.Services.ExampleService`.

---

### `StarterPlayer.StarterPlayerScripts.Client`

```json
"Client": { "$path": "src/startup/Client.client.luau" }
```

The client-side entry point script. Rojo syncs it as a `LocalScript` (because of the `.client.luau` suffix). Like the server entry point, this is a static node and `src/startup` is blacklisted from the walk.

<Info>
  The naming convention for entry point scripts (`.server.luau` and
  `.client.luau`) is a Rojo convention, not a genRojoTree convention. Rojo uses
  the suffix to determine the Roblox script type when syncing.
</Info>
