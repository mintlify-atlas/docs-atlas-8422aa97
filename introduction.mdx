Maintaining a `default.project.json` by hand is tedious and error-prone. Every time you add, move, or rename a Luau script, you have to remember to update your Rojo project file â€” or your sync breaks. genRojoTree eliminates that toil by reading your `src/` folder directly and generating the file for you, using a set of predictable naming conventions to decide where each script belongs in the Roblox service tree.

<Info>
  genRojoTree is designed for use with **Rojo 7.5.1** (managed via [Rokit](https://github.com/rojo-rbx/rokit)) and requires **Node.js** to run.
</Info>

## What problem it solves

Rojo maps filesystem paths to Roblox service locations through `default.project.json`. In a growing project, that file can easily have dozens of entries â€” and keeping it in sync with your `src/` folder is a constant maintenance burden.

genRojoTree automates this by walking `src/` at build time, applying naming conventions to determine service targets (`ReplicatedStorage` vs `ServerScriptService`), and writing a fresh `default.project.json` on every run. The result is a project file that always reflects your actual source layout.

## Key features

- **Auto-generation** â€” run `npm run build:rojo` and `default.project.json` is recreated from scratch based on what is in `src/`.
- **Naming conventions** â€” files containing `server` in their name route to `ServerScriptService`; all others go to `ReplicatedStorage.Shared`.
- **`init.luau` promotion** â€” a folder containing `init.luau` is promoted to a module in-place, with its children suppressed to avoid duplication.
- **Blacklist support** â€” directories like `src/ui` and `src/startup` are excluded from the walk and handled with static `$path` entries instead.
- **Watch mode** â€” `npm run watch:rojo` uses `chokidar` to re-run generation whenever any file under `src/` changes.

<Note>
  genRojoTree outputs a single file: `default.project.json`. This file is what Rojo reads when you run `rojo serve` or `rojo build`.
</Note>
